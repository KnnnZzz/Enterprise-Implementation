# Day 4 — MISP Threat Intelligence Integration

> **Goal:** MISP connected for active IoC matching, CDB blocklists populated and auto-synced, every IP/hash/domain checked against threat feeds.  
> **Estimated time:** 6–8 hours

---

## 4.1 Prerequisites

- A running MISP instance (self-hosted or community access)
- MISP API key with `read` + `sighting` permissions
- MISP CA certificate (if self-signed)
- `requests` Python library for Wazuh's embedded Python

```bash
# Install requests in Wazuh's Python environment
/var/ossec/framework/python/bin/pip3 install requests

# Store API key securely (NOT in ossec.conf)
echo "YOUR_MISP_API_KEY_HERE" > /var/ossec/etc/misp.secret
chmod 400 /var/ossec/etc/misp.secret
chown root:wazuh /var/ossec/etc/misp.secret

# Copy MISP CA certificate (if self-signed)
cp /path/to/misp-ca.pem /var/ossec/etc/misp-cert.pem
chmod 444 /var/ossec/etc/misp-cert.pem
```

## 4.2 Deploy the MISP Integration Script

This script intercepts Wazuh alerts, extracts IoCs (hashes, IPs, domains, registry keys, filenames), and queries MISP in real-time.

```bash
# Copy the integration script to Wazuh
cp custom-misp_file_hashes.py /var/ossec/integrations/custom-misp_file_hashes.py
chmod 750 /var/ossec/integrations/custom-misp_file_hashes.py
chown root:wazuh /var/ossec/integrations/custom-misp_file_hashes.py
```

### Key Features of the Script

| Feature | Description |
|---|---|
| **Multi-IoC extraction** | Hashes (MD5/SHA1/SHA256), IPs, domains, registry keys, filenames |
| **Loop prevention** | Ignores alerts generated by MISP rules (100800–100810) |
| **Private IP filtering** | Skips RFC1918 addresses to avoid wasting API calls |
| **Whitelist** | Configurable list of known-good IoCs to skip |
| **Sightings push** | Reports matches back to MISP for community intelligence |
| **Secure API key** | Reads from `/var/ossec/etc/misp.secret` instead of ossec.conf |
| **Active Response** | Injects `srcip` and `syscheck.path` for automated blocking |

### Add Integration to `ossec.conf`

```xml
<!-- Add to /var/ossec/etc/ossec.conf -->
<integration>
    <name>custom-misp_file_hashes.py</name>
    <hook_url>https://YOUR_MISP_URL</hook_url>
    <api_key>LOADED_FROM_SECURE_FILE</api_key>
    <level>5</level>
    <alert_format>json</alert_format>
    <options>{"timeout": 10, "retries": 3, "debug": false,
              "tags": ["tlp:white", "tlp:clear", "malware"],
              "push_sightings": true,
              "skip_private_ips": true,
              "ca_cert": "/var/ossec/etc/misp-cert.pem"}</options>
</integration>
```

> [!WARNING]
> **`<level>5</level>`** means the script runs on ALL alerts level 5+. For a small company this is fine. If your MISP instance is slow or rate-limited, increase to `<level>8</level>` to reduce API calls.

> **⚠️ PITFALL — `api_key` field:**  
> The `api_key` field in ossec.conf is **still required** by Wazuh's integration framework (it passes it as an argument). Set it to any placeholder — the script overrides it by reading from `/var/ossec/etc/misp.secret`. If you leave `<api_key>` empty, the integration won't start.

## 4.3 Deploy MISP Alert Rules

Create `/var/ossec/etc/rules/misp_rules.xml`:

```xml
<group name="misp,malware,">

    <!-- Base MISP integration event -->
    <rule id="100800" level="0">
        <decoded_as>json</decoded_as>
        <field name="integration">misp_file_hashes</field>
        <description>MISP: IoC check event.</description>
    </rule>

    <!-- No match found — silent -->
    <rule id="100801" level="0">
        <if_sid>100800</if_sid>
        <field name="misp_file_hashes.found">0</field>
        <description>MISP: IoC not found in threat feeds.</description>
    </rule>

    <!-- MATCH FOUND — Generic fallback (Level 14) -->
    <rule id="100802" level="14">
        <if_sid>100800</if_sid>
        <field name="misp_file_hashes.found">1</field>
        <description>MISP Threat Intel: Confirmed Detection — $(misp_file_hashes.value)!</description>
        <mitre>
            <id>T1588</id>
            <id>T1105</id>
        </mitre>
        <group>misp,threat_intel,</group>
    </rule>

    <!-- Specific IoC types for better SOC triage -->

    <!-- Malicious file hash (Level 15 — highest priority) -->
    <rule id="100806" level="15">
        <if_sid>100802</if_sid>
        <field name="misp_file_hashes.source.ioc_type">hash</field>
        <description>MISP: Malicious file hash confirmed — $(misp_file_hashes.value)</description>
        <mitre>
            <id>T1204.002</id>
        </mitre>
        <group>misp,threat_intel,malware,</group>
    </rule>

    <!-- Malicious IP communication -->
    <rule id="100807" level="14">
        <if_sid>100802</if_sid>
        <field name="misp_file_hashes.source.ioc_type">ip</field>
        <description>MISP: Communication with known malicious IP — $(misp_file_hashes.value)</description>
        <mitre>
            <id>T1071</id>
        </mitre>
        <group>misp,threat_intel,</group>
    </rule>

    <!-- Malicious domain resolution -->
    <rule id="100808" level="14">
        <if_sid>100802</if_sid>
        <field name="misp_file_hashes.source.ioc_type">domain</field>
        <description>MISP: DNS query to known malicious domain — $(misp_file_hashes.value)</description>
        <mitre>
            <id>T1071.004</id>
        </mitre>
        <group>misp,threat_intel,</group>
    </rule>

    <!-- Registry key match -->
    <rule id="100809" level="13">
        <if_sid>100802</if_sid>
        <field name="misp_file_hashes.source.ioc_type">regkey</field>
        <description>MISP: Malicious registry key detected — $(misp_file_hashes.value)</description>
        <mitre>
            <id>T1112</id>
            <id>T1547.001</id>
        </mitre>
        <group>misp,threat_intel,</group>
    </rule>

    <!-- Malicious filename -->
    <rule id="100810" level="12">
        <if_sid>100802</if_sid>
        <field name="misp_file_hashes.source.ioc_type">filename</field>
        <description>MISP: Known malicious filename detected — $(misp_file_hashes.value)</description>
        <mitre>
            <id>T1036</id>
        </mitre>
        <group>misp,threat_intel,</group>
    </rule>

    <!-- Error rules for integration health monitoring -->
    <rule id="100803" level="10">
        <if_sid>100800</if_sid>
        <field name="misp_file_hashes.error">403</field>
        <description>MISP ERROR: Invalid API credentials.</description>
    </rule>

    <rule id="100804" level="10">
        <if_sid>100800</if_sid>
        <field name="misp_file_hashes.error">429</field>
        <description>MISP ERROR: Rate limit exceeded.</description>
    </rule>

    <rule id="100805" level="10">
        <if_sid>100800</if_sid>
        <field name="misp_file_hashes.error">500</field>
        <description>MISP ERROR: Server error — $(misp_file_hashes.description)</description>
    </rule>

</group>
```

## 4.4 CDB Lists — Manual & Automated Blocklists

### Understanding CDB Lists

CDB (Constant Database) lists are key:value flat files that Wazuh rules can look up in real-time. Format: `indicator:category`

### Create the Directory Structure

```bash
mkdir -p /var/ossec/etc/lists/malicious-ioc
```

### Seed Initial Lists

```bash
# Malicious IPs
cat > /var/ossec/etc/lists/malicious-ioc/malicious-ip << 'EOF'
185.220.101.1:tor_exit_node
45.33.32.156:known_scanner
EOF

# Malicious domains
cat > /var/ossec/etc/lists/malicious-ioc/malicious-domains << 'EOF'
evil-domain.com:phishing
malware-c2.net:command_and_control
EOF

# Malware hashes
cat > /var/ossec/etc/lists/malicious-ioc/malware-hashes << 'EOF'
44d88612fea8a8f36de82e1278abb02f:eicar_test
EOF

# Trusted IPs (whitelist — never trigger alerts)
cat > /var/ossec/etc/lists/trusted-ips << 'EOF'
8.8.8.8:google_dns
1.1.1.1:cloudflare_dns
YOUR_PUBLIC_IP:company_egress
EOF
```

### Register CDB Lists in `ossec.conf`

Add to the `<ruleset>` section:

```xml
<ruleset>
    <!-- ... existing entries ... -->
    <list>etc/lists/malicious-ioc/malicious-ip</list>
    <list>etc/lists/malicious-ioc/malicious-domains</list>
    <list>etc/lists/malicious-ioc/malware-hashes</list>
    <list>etc/lists/trusted-ips</list>
    <!-- ... -->
</ruleset>
```

### CDB Lookup Rules

Add to your `local_rules.xml` or `misp_rules.xml`:

```xml
<!-- Alert if source IP is in malicious CDB list -->
<rule id="100500" level="13">
    <if_sid>5710,5711</if_sid>
    <list field="srcip" lookup="address_match_key">etc/lists/malicious-ioc/malicious-ip</list>
    <description>CDB: Connection from known malicious IP $(srcip).</description>
    <group>threat_intel,cdb_match,</group>
</rule>

<!-- Alert if FIM hash matches malware hash list -->
<rule id="100501" level="14">
    <if_sid>550,554</if_sid>
    <list field="sha256" lookup="match_key">etc/lists/malicious-ioc/malware-hashes</list>
    <description>CDB: File hash matches known malware — $(sha256)</description>
    <group>threat_intel,malware,cdb_match,</group>
</rule>
```

## 4.5 Automated CDB Sync from MISP (Cron)

Deploy the batch sync script that pulls IoCs from MISP and populates CDB lists:

```bash
# Copy sync script
cp misp-cdb-sync.py /var/ossec/integrations/misp-cdb-sync.py
chmod 750 /var/ossec/integrations/misp-cdb-sync.py
chown root:wazuh /var/ossec/integrations/misp-cdb-sync.py

# Schedule: run every 4 hours
cat > /etc/cron.d/wazuh-misp-sync << 'EOF'
0 */4 * * * root /var/ossec/framework/python/bin/python3 /var/ossec/integrations/misp-cdb-sync.py >> /var/ossec/logs/misp-cdb-sync.log 2>&1
EOF

# Run it manually the first time
/var/ossec/framework/python/bin/python3 /var/ossec/integrations/misp-cdb-sync.py

# Check results
wc -l /var/ossec/etc/lists/malicious-ioc/*
cat /var/ossec/logs/misp-cdb-sync.log
```

The sync script:
- Fetches IoCs from MISP (last 90 days, `to_ids=1`, published only)
- Filters by type: `ip-src`, `ip-dst`, `domain`, `hostname`, `md5`, `sha1`, `sha256`
- Validates IPv4 format and strips invalid entries
- Writes atomic CDB files (write to `.tmp`, then `rename`)

> **⚠️ PITFALL — CDB lists require manager restart:**  
> After CDB list files are updated, Wazuh needs a restart to reload them into memory. The cron job updates the files, but you need to either:
> 1. Add `systemctl restart wazuh-manager` to the cron (adds ~10s downtime)
> 2. Or accept a delay until next natural restart
> For a small company, option 1 is acceptable.

## 4.6 Optional: Active Response on MISP Match

Enable automatic IP blocking when MISP detects a threat:

```xml
<!-- In ossec.conf -->
<active-response>
    <disabled>no</disabled>
    <command>firewall-drop</command>
    <location>local</location>
    <rules_id>100802,100807</rules_id>  <!-- MISP generic + IP match -->
    <timeout>3600</timeout>             <!-- Block for 1 hour -->
</active-response>
```

> [!CAUTION]
> **Test active response carefully before enabling in production.** A false positive in MISP can block legitimate traffic. Start with `<disabled>yes</disabled>` and monitor MISP matches for a few days before enabling automated blocking.
